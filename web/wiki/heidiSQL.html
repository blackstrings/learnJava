HeidiSQL

A client SQL tool we use for reading and writing data related to our design-it applications

Our applications constantly pull data from our mySQL servers for image paths, SKU, and string data. Our applications need access to our mySQL server to read data such as things like string paths locations for our images, so that our applications can pull the correct image paths from within the application's workspace and display the image on the kiosk

    Database Relationships

    • one to one:
      • product has a product description specifically about that product
      • when you have too much column in one table, you can split up the table to ref another table, used for optionals cols
    • one to many:
      • mother can give birth to many children, but not the other way around
    • many to many:
      • A parent can have many children, one children can have or belong to many parents
      • author can write many books, one book can be written by many authors
      • customer can have many addresses, one address can have or be shared by many customers
      • when this occurs, you want to have a junction or intermediate table to handle and connect the two tables

    Reference tables are used in environments where one table relies on a code rather than an id. Where a code can represent a defined number of possible seletions. Take for example, order status. You would have ORDER table and REF_ORDER_STATUS table. ORDER table would have a code that reference the REF_ORDER_STATUS.
