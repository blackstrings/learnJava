JUnit

A java third party tool that comes with eclipse (can be manually installed to other IDE) that we can use to test our java codes in whatever project we work in. We use JUnit for testing so that we can see, hit and ensure all lines in a particular function is working. We don't have to test getters and setters.

It is a quick way to test our code without opening or running the application GUI itself. JUnit will run the test class and give instant feedbacks if a certain function failed or passes. The idea is, you write a test to test each function, and in the future after code changes to your app, you run the all the test and it will tell you if the new codes broke any of your old functionalities.

Setup

  • To start writing Junits, you have to have a Junit MCR created for you first, the MCR is usually set to 40 hours
  • Junits are good to work on during down times when most MCRs are done
  • When you get an MCR for Junit, you will be focused on a specific application and also under a specific package
  • To test your codes, you first have to study and understand the package and the app, and what you will be testing specifically under each package
  • Use the @test anontation above each of your test functions
  • Assert.assertTrue( test, answerWeAreLookingFor );

What you want to avoid testing

  • Methods inside methods
  • codes with no logic (or codes that do not have if and then statements)

Key purposes of JUnit

  • The sole purpose of writing a test method for the class that you are testing, is to make sure that you eventually call a class's method within the scope of the test method. Not calling a single class's method at all defeats the purpose of Junit
  • JUnit testing doesn't care about private methods because it shouldn't need to, if you must however, use PowerMockito
  • Perform unique individual code specific test or as it says, unit testing
