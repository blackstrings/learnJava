Environment Pointing

All of our kiosk applications point to an environment when running the application. They point to environments so that they can get the correct data when loading product information (ie. type, size, length, name, etc) -- not pricing though, as pricing is pulled from a different data not related to environment pointing

When testing or running the applications at our desk PC, we try not to run the kiosks under production (prod) environment -- just something the business team prefer us not to do. So instead we point to staging environment. From time to time, you'll see that you may have to change the environment for some applications when running certain test in the lab or at your computer. Recently our developer created a EvnSetter tool which lets use change the config without opening any files.

Take note that depending on what the application it is (ie. Java, C++, webapp) there is a different method on how to go about changing the environment for each.

C++

As of Dec 2015, C++ apps now use the reaf environments, before the CApps didn't use the reaf environments

  • Because all the product data are located inside the ini files within the app root folder, so there is no need to point to environments for C apps
  • However, every app does point to a store regardless of why type of app it is
  • C:/ > menards > utils > host.ini & store.ini
  • 4444 is our test store which usually is correct, but you can point to 3011 (store east) if certain products aren't showing up
  • Other times when ask to point to staging with C apps, it may just mean to get or test the staging branch from the repository using SVN. In other words, just checkout (svn) the staging branch as you would with any other C app for testing.
