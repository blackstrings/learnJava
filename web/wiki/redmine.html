Redmine

A third-party online site we use to track issues and work on as our primary everyday tasks. It helps us manage, maintain, and track down old, new, and current tickets. It is a process that organizes our tickets from the Menard business team -- they make the tickets, we work on it..

This is how we monitor our time spent working here at Menards in the IS department. Aside from clocking in the moment we get into our work office, we also have to clock/time stamp each ticket that we work on. If we are helping a team mate or working

Setup

  • You'll need a login to access the Menard Remine.
  • http://redmine.menards.net/login

CHEAT SHEET for learning the Whole Redmine Process for Newbies

Discovery (find out how much hours it may take to complete this MCR)

Eclipse

Talk to a primary or secondary to know where to begin if not sure
Download from SVN the staging branch of the app
Try to figure out what is needed to make these changes
Get a rough estimate and send the ticket back to the Team lead with description if any and estimated hour

Business Design (Create the Business Design document)

Microsoft Word

Usually you can find the info and requirements in the Redmine Ticket to put on the BD
  • Get the BD template named something like "Design#####-example.doc" and modify it
  • The title on doc is same as Redmine ticket
  • Login into Redmine > get to the ticket > look at description
  • Copy & Paste description to the business requirement section
  • Add a screen shots of current and future if applicable
  • Delete stuff from the template that doesn't apply
  • Save BD as "DICDesignMCR_NUMBER_GOES_HERE.doc"
  • Save > attach BD doc to the redmine ticket and send it back to team lead under the same status Business Design

There are stages before the Test Design, but newbies usually will start here at this stage

Test Design (creating the test scripts, cases, suite in RQM)

Redmine

Login & View Assigned Redmine Ticket, you will have one when you start your first redmine ticket
Record Start Time for this MCR into excel (optional)
Check category section to know what app you will be working in
Open/Read attached Business Design (BD) word doc
Understand what you will need to fix in the BD (Business description on page 2)
  • Sometimes the description may or may not be detailed enough to make the responsible programmer (you) understand what the business team whants you to fix
  • If so, ask for help by other team members or teamlead, or worse case, you'll have to send back the MCR to the business team to further correct the description

Eclipse || Visual Studio 6 || Heidi || ETC (this step can be skipped)


If needed, prepare to do some simple application test if possible to understand the BD and causes
Update the app workspace to the latest staging using SVN before testing
Perform necessary test untill you understand what the BD is asking you to fix

RQM (once you get how to test the design)

Create Test Script
  • Login into RQM > Construction > Create > Test Script
  • Name = Redmine MCR "subject" value
  • Owner = Primary/Secondary
  • Team Area = IDS - Design IT Team
  • Save
Open Application at your computer with the right IDE (Depends on business design, you may or may not do this)
Add the necessary Steps to run through the app to cover the business design requirements
  • To know requirements to cover, you have to look carefully at what the business design wants
  • With experience, you'll start to know what to test for
  • Every action you click on the app screen should be recorded as a step in RQM as a step
  • Expected results should be enter when you expect something that will full fill what is described in the BD
Create test case, test suite
  • Test script: same name as MCR
  • Team Area: IDS Design-It Team
  • Owner: primary
  • IS-System: Design-it
  • IS-System-Application: Design-it Deck
  • save
  • Test case: same name as test script
  • Team Area: IDS Design-It Team
  • Owner: primary
  • IS-System: Design-it
  • IS-System-Application: Design-it Deck
  • save
  • Create Test Suite
  • Name: MCR-[MCR#] ex: MCR-129949
  • Team area: IDS Design-It Team
  • Owner: primary person
  • IS-system: DesignIt
  • IS-System-Application: Internal
  • save
Assign test script to test case, test case to test suite (search by id for faster locating)

Redmine

Edit MCR and update to Test Design
  • Assign to: unchanged (should still be your name)
  • Status: Test Design to Tech Design
  • RQM: Name of the test suite created in RQM or ex: MCR-124494
  • Description > edit > copy/paste Test Suite URL into bottom of description
  • Log Time -- Activity: Design
  • Spent Time: example for 1hr 30 min put 1.5
  • Comment: Test Design
  • Submit

Tech Design (Modify BD word doc to instruct what you will be doing)

Business Design (BD)

Open and modify BD
  • Record Start Time in excel (optional)
  • Open and Save As to new file name: insert "Tech" into the file name
  • ex: DICDesign122834.docx >> DICTechDesign122834.docx
  • Make sure you put the instructions in the correct section within the document
  • Ask for help if not sure which section to put in
  • Basically these specific instructions should allow anyone to make the changes for you if you are absent for work
  • Copy and paste code as needed if the code section is small enough
  • Highlight Old from New changes within the codes if pasting codes
  • If complicated changes are going to be made, ask mentor or teamlead for help on how to write these instructions
  • Put n/a under DIC Engineering Book section if not adding any new functions
  • When done, Save

Redmine

Edit MCR and update to Test Design Review
  • Assign to: Primary/Secondary
  • Attach the modify tech BD
  • If there is SQL data changed, attached that template too
  • Log Time
  • Activity: Design
  • Comment: Tech Design
  • Submit
  • Wait for assigned person to send you back the MCR in construction status (you'll get an email)

Construction (Request branch & apply actual changes to codes or data)

Outlook

Check status if MCR is in construction, if so proceed, if not check MCR notes on what went wrong
Email release managers for a branch, if MCR is not mySQL data related
  • email: dirm@menards.net
  • subject: Feature Branch needed
  • Msg: Can I please get a feature branch for [ApplicationName] + [Redmine#] + [Release Date, check MCR target version date to know the Month or release quarter] ex: Deck Redmine#233234 Q3R1
  • send email and wait, when you get an email back saying feature branch has been created, proceed

Eclipse (if Java App, otherwise skip)

Create a new folder in your workspace named something like "workspace+[MCR#]" ex: workspace122834
SVN only the feature branch into the created workspace, and create these framework folders (3D, Common, Reaf)
Close all three framework projects inside eclipse except the app (right-click > close proj)
Right click and run the app.launch file to see if it works, if so, make changes stated in BD and test till things work

SVN


When app runs properly after code changes > window explorer > rightclick on folder or specific files > Tortoise SVN > check for modification > will run Wintail (if install) to verify code changes between original and new codes
SVN commit just the feature branch folder back if all changes are good
  • Comment: refs #[redmine#] then say whatever you did
  • Click OK (not merge)

Visual Studio 6 (if C++ App, otherwise skip)


Redmine

Edit and Update MCR and submit
  • Edit MCR
  • Assign to: unchanged (should still be you)
  • status: Construction to Code Review
  • % done: to 100%
  • If SQL changes, Change Production Data Changes to Yes
  • Function Releae Notes: [desc for the business team who created ticket] ex: Show correct image on button if pressed
  • Technical Release Notes: [desc for the coders] ex: image source tag change from buttons/img1.png to buttons/img2.png
  • Log time for construction
  • Comment: Construction
  • submit

Code Review (upload and allow other team members to review your code changes)

Code Collaborator (CC) Client tool

Record Start time in excel (optional)
Install the client CC if not have > run setup > open
Prepare to upload feature branch to Code Colaborator (can be done two ways, using the client CC tool or the web CC browser -- client tool is faster but web CC is easier)

Method 1
TortoiseSVN - Using the CC web browser (easier but slightly slower)

  • After you commit your feature branch to the Repo using tortoiseSVN
  • CC browser > create new review
  • Title: MCR# + [Redmine Ticket Name]
  • Overview: subject of MCR
  • Restrict Access: Anyone
  • Restrict uploads: No
  • Deadline: leave default
  • Set Author (you), at least 2 Reviewers (Primary, Secondary, Tertery), include everyone else as Observers
  • Attach Checked-in Changelist
  • Middle mouse click "Apply and Continue Review" to see a temp page
  • After verify everything is correct > apply and begin review

Method 1 cont. - Redmine

Copy CC review # or URL to MCR description > submit


Method 2 - Using CC Client Tool (more advance)

  • Add subversion diffs
  • Title: MCR# + [Redmine Ticket Name]
  • Next
  • Branches/Tags tab
  • Before Sub. URL: SVN staging url
  • After Subversion URL: your feature branch SVN url

Method 2 cont. - TortoiseSVN

  • Open explorer window and locate your workspace
  • Right-click > tortoiseSVN that explorer window > Repo Browser
  • go to your feature branch > right click > show log
  • Click finish when done and CC browser will launch allowing you to review and assign reviewers before you launch

Method 2 cont. - Code Collaborator (browser)

Assign Reviewers (they are primary, secondary, and tertery and then observers if want
  • After assigning wait for all reviews to pass
  • if not pass, make necessary changes until pass

Redmine

Everyone will get emailed automatically, When all reviewers pass the review, assign MCR to Merica and submit no need to put any notes

Testing (Test RQM test cases in Lab Room on specified kiosks machine)

Redmine

Look up MCR, find RQM ID

RQM

login, and Look up RQM test case #ID
Run test cases in parallel and pass them if they pass, if fail talk to person who created the RQM first before failing it

Redmine

Edit MCR, Assign to Merica, Status:Automated Testing, note: testing pass, Logtime
