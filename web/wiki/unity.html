Good Practices

Unity is built around components, so it is best to also build your scripts like components. Think modules. One script specifically does one task. Say you have a hero game object with hp, weapon, and inputs. You'd create 3 classes total to handle each of these components, and you put it on the gameobject

Also think, interface. Build your classes around interfaces. For example, HP class should implement a Interface IDamagable if HP class is going to handle damaging system. HP class should also implement a 2nd, 3rd interface or more, if it is handling other functionalities that can be grouped as a category. Class A shouldn't have to access directly methods in Class B, but rather access an interface that class B has implemented.

What to use: MonoBehaviour vs ScriptableObject vs Normal

Monobehaviour

  • When you want to attach to gameobjects, use unity editor gui, tweak public variables
  • Allows you to utilize Start, Awake, Update, OnMouse...etc

ScriptableObject

  • When you only need to manage data and will need class to be serialize for saving
  • Reason is because Unity uses special serialization unique only to Unity
  • Avoid passing parameters to constructor, best to leave parameters empty, create init method if need to use parameter
  • to create an instance using new
  • instead do MyClass myClass = ScriptableObject.createInstance<MyClass>();
  • Then follow up with myClass.init() if you wish to run a constructor on it, but don't create a true constructor for it
  • ScriptableObject aren't meant to have a constructor the way it is built

Normal classes

  • The class does not need to be saved or serialized
  • The class does not get attached to game objects
  • The class is not referenced in a another class that will be serialized.
  • Good for utility classes

Network Communication

You have two options, get using WWW alone or post using WWWForm combined with WWW. To use the class, the instantiating class can't be static due to monobehaviour. So create a class that can last throughout the app if you are doing scene loading. If you do not make a singleton class of this web call, when you load a scene, it will stop/interrupt the current communication during a load scene.

Webplayer and webGL builds using server calls must have a crossdomainpolicy.xml set at the root level of their domain within the server. Helpful links Web comm1. If you do this however, be extra careful that only the correct game calls can talk to the server as doing this opens the portal for outside attacks.

However, if the game is loaded from the same web server the data is in, this cross domain issue will not occur and will load data fine.

WebGL Builds

This build usually creates 30MB+ package folder due to all the c# conversion to javascript.

  • You need to have "write" access permission to your programFiles/Unity folder (contact DesktopSupport)
    • webGL builds some files to this very folder at buildtime
  • Build Setting Configs
    • Unity 5.2+ - Build without having anything selected, not even dev mode - auto optimization in place
    • Unity 5.1 - Build with fast build, stripping may be needed
  • When running webGL builds on your local machine, and unity is making a call to another server (stage, prod) on another machine, you need to Allow the server to open security for cross-domain-policy, put this file on the root directory of server
    • <?xml version="1.0"?>
    <cross-domain-policy> <allow-access-from domain="*"/>
    <allow-access-from secure="false"/>
    </cross-domain-policy>
  • The alternative to this is, while making local builds, run your own local server so you don't have to worry about the crossdomainpolicy.xml file.
  • If build breaks, try increasing memory to 512 and enable exceptions to full in public settings
  • Avoid try catch statements, webGL seems to break in the try catch. If you want to use try catch, you'll have to enable debug mode so that web GL builds the exception package to the build, which is only good for debugging purposes.
