RQM - Lab Testing and how it works

Lab testing is when we go to the lab room and run RQM/MCR test or regression test cases on the Beast machine. This process requires us to move constantly back and forth between the Beast machine and the Kiosk nearby as we perform the test cases. We don't really need to use the Beast Machine, but it is the closes PC that has access to the internet or the RQM online site where we can record our test cases results.

At Menards we need to be able to test and simulate kiosks located at an actual Menard store properly, so we need to have these kiosks PC machines in handy to test against. So, in the lab we have kiosks that are built just like the ones in the store. Each Kiosk in the lab is assigned a test store number.

For example, the Menard store in Wausau is code numbered 3016, Eau Claire East is 3053, Eau Claire West is 3011 and so forth as each store has a unique 4 digit number. All kiosk computers located in each store will be based off their store number and so when orders are made from a paticular store, the processed order will remember the store number and the kiosk # the order was from. Which then, POS (department) will receive that paticular order and will know which store that order came from and which kiosk.

But because our Kiosks are in a lab environment for testing purposes and are not in an actual store, you may ask how do we simulate a kiosk at an actual store then. Simply, we use a test store number instead which don't represent a physical store but a fake store. There are multiple test store numbers. These test store numbers are 4444, 5555, 6666, 8888, 9696. So the computers in the lab are label with these test store numbers -- confusing? if so, imagine that these test store exist only in our dreams and are not physical stores like Eau Claire East and West. When we make orders on our test kiosk, they will still show up under POS, but will be from the test stores and POS will ignore these orders.

So Really, the kiosks are just computers, either XP or Windows 7. Each kiosk represent or points to our fake store number for testing purposes. Our test store (computers like 8888) will simulate staging (a phase where where we manually install our apps onto these computers if we need to perform testing). 5555 is our production test store computers where we don't manually install our test apps onto but instead another team uses the software Altiris to perform a remote install of our apps onto these computers during deployment test release phases. When all test have passed on the 5555, then Altiris will release the passed final versions onto the 4444, which are our test stores. Whatever we see on the 4444 computers, is what we will also see at an actual physical store. The imaging team then will send auto updates to the store at East and then at West for finalizing our test. Once our apps passed at those store, the imaging team will gradually send auto install to all the stores nation wide.

You will use the computer in the lab to do the testing (usually the beast) as we want to simulate a kiosk that is at a store. You will use one screen for RQM while the other kiosk screen to perform your test.

You don't actually have to use the beast computer, and can use your office computer, but the beast is the closest to the kiosks that can access RQM

Tecnical - Pausing and resuming test cases in RQM

  • To pause a script: click pause button
  • To resume a script: find and open the suite or test case you have started. Look under Execution Records and click on the paused icon to resume.

Designit Apps for both 4444 and 8888

When we test, we have to make sure we are pointed to the appropriate environment and the case often switches depending on the test case you are running. First thing and foremost, Make sure to override the store service id in the envrionment, change it to 3011. Location of the file is under c:\menards\env\DesignItConfig\VERSION\designItConfigData.properties

If you are testing a regular MCR, most of the time your environment should be pointed to env=stage.ua-designit.store and not internal.store. Unless otherwise mentioned in the test case to use a specific environmnet, keep the environment pointed to stage.

For regression testing, you often would want to point your environment to prod.main.store unless otherwise specified by the test case.

Siding, Roofing, Gutter

These 3 are a little different when testing these app. You have to make sure your environment is pointing to kiosk-stage in order to load in products when using them. Follow instruction below to do this:

8888 Lab Computers

  • On 8888 Test Lab Computer > F6 > enter pass > Explorer > Menards > Utils > MenardAppMenu > dsgnit.xml
  • Scroll to the bottom and you should see a set of 3 similar lines
  • Change the argument to look like below if you see "designit.mymenards" anywhere in the argument
  • The AppArg should be: (note the & should be spelled out with its caracter code)
  • AppArg = http://stage-kiosk.mymenards.com/..../sidingEstimator/screen1100.do?dispatch=restart&selection(storeNumer)=3011"
  • for prod use = http://design-it.mymenards.com/..../(storeNumber)=3011
  • Keyword is "stage-kiosk" and &selection(storeNumber)=3011

4444 Lab Computers

  • On 8888 Test Lab Computer > F6 > enter pass > Explorer > Menards > Utils > MenardAppMenu > dsgnit.xml
  • Scroll to the bottom and you should see a set of 3 similar lines
  • The AppArg should be: (note the & should be spelled out with its caracter code)
  • AppArg = http://stage-kiosk.mymenards.com/..../sidingEstimator/screen1100.do?dispatch=restart&selection(storeNumer)=3011"

Gutter speical exception

  • For Gutter, the line change is towards the top where the button tag is with the rest of the other apps

How to know if app is the latest testing RC

For C++ apps, check under the c/mes or somefolder (talk to c++ app users). For java, check under c\mendards\ and corresponding app jar date. Their date should be close to your testing date. If not, you may not be testing the latest RC.
